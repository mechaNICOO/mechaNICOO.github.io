用户的输入与3D模型的互动

所有的输入检测都是通过QObjectPicker。QObjectPicker的原理是跟踪图像射线的轨迹，检测出轨迹与物体的干涉（？？？）。算法和光线追踪相似，但是我们不需要知道被指向的物体的颜色、材质或者光线与其相交的点，我们只需要知道物体和射线相交了。

在QEntity中添加QObjectPicker就可以使用一些信号了。例如pressed(),released()和clicked()。对于一些拖拽的事件，确保dragEnabled的值为true，然后使用moved()信号。

所有的信号都提供了一个QPickEvent对象，这个对象使用position（相对屏幕）或localIntersection（相对模型）或worldIntersection（相对世界）坐标系来描述（不太理解是什么意思）

ECS架构

  • Entity实体:
    实体只是一个概念上的定义，指的是存在你游戏世界中的一个独特物体，是一系列组件的集合。为了方便区分不同的实体，在代码层面上一般用一个ID来进行表示。所有组成这个实体的组件将会被这个ID标记，从而明确哪些组件属于该实体。由于其是一系列组件的集合，因此完全可以在运行时动态地为实体增加一个新的组件或是将组件从实体中移除。比如，玩家实体因为某些原因（可能陷入昏迷）而丧失了移动能力，只需简单地将移动组件从该实体身上移除，便可以达到无法移动的效果了。
  • Component 组件
    仅仅包含代表其特性的数据（即在组件中没有任何方法）
  • System系统
    其只拥有行为（即在系统中没有任何数据）

QML中的Entity类型

Entity本身是什么都没有的。需要通过components为其添加各种实现功能的组件。

QML中的Transform类型

用来表现模型的形变、旋转。
大概就是绕各种轴的旋转、四元数的解码、模型大小缩放之类的。

QML中的RenderSettings

RenderSettings类型用来储存渲染处理的设置和起作用的FrameGraph
RenderSettings组件必须在场景的root entity中设置。

关于 Render Framegraph

framegraph是控制相关的数据结构。和ECS结构一样，framegraph也是一种树结构，只不过用作于场景的渲染。在渲染帧的过程中，3D渲染器会不断的改变state。state的数量和性质不仅取决于materials（例如shaders,mesh geometry），也取决于我们选择的更高层次的渲染策略、计划。

关于inputSettings

inputSettings保存了指向输入事件源(input event source)对象的指针。
inputSettings组件必须在场景的root entity中设置。

插件的URI

详见Identified Modules这篇文章，URI名字不能随便来，URI中的“点”是用作文件分级的。

DESTDIR +=

在制作插件时，插件名.pro文件中添加DESTDIR += 想要dll文件导出的目录，这样在编译插件工程时会自动将dll文件导入到指定目录。

手动复制qmldir文件

可能是我的设置原因（我还不知道怎么设置能让qmldir文件自动出现在它应该出现的地方），qmldir文件总是会丢失、或者不产生。

首先是在编译插件时，提示找不到qmldir文件。这时候把qmldir文件拷贝到buld-xxx-MinGW_32bit-Debug（取决与平台和编译器）中即可。

然后是编译结束后，要把qmldir文件和dll文件放在一起。

Identified Modules

Identified Modules 时通过一连串带点号的URI被QML引擎识别的。

Locally Installed Identified Modules

包含qmldir文件并添加金QML import path的QML或C++文件可以被用作Identified Modules.

qmldir文件一定要在import path的目录结构中，并反应出URI的identifier字符串，在URI中的每一个“.”都表示目录中的一级。例如com.mycompany.mymodule中的qmldir文件，一定是在import path中的com/mycompany/mymodule/qmldir中。

一个module的不同版本是可以保存在同一个目录的子目录中的。例如module的2.1版本可以在com/mycompany/mymodule.2.1/qmldir中。

当然一个module的不同版本也可以在qmldir文件中直接定义，但是这样会使更新module变得有点麻烦。
